## 项目：一人公司 · 宇宙版 (one_company_cosmic)

### 目录结构
```
core/          ← 核心层，严禁修改
  cosmic.py    — 深空渲染引擎（背景/星云/粒子/辉光）
  agent.py     — 智能体引擎
  data.py      — 数据层
  llm_client.py— LLM 客户端
  voice.py     — 语音引擎

modules/       ← 业务模块，只新增不改已有
  auth/        — 登录/注册
  dashboard/   — 主面板
  business/    — 客户/订单/产品/财务
  personnel/   — 员工/会员/分润/钱包
  intelligence/— AI中心/聊天/数智员工/扫描/工具箱
  data_center/ — BI看板/报表
  system/      — 激活/云同步/日志

main.py        — 入口，不改
gen_book.py    — 文档生成器，不改
rollback_control.py — 回滚管控，不改

项目全书/      ← 项目文档（33 个分片文件，每章独立）
  README.md    — 总索引
  模块状态总表.md — 54 个模块行数/状态
  版本功能地图.md — 变更记录时间线
  回滚管控策略.md — 回滚铁律与流程
  卷01_核心引擎/ — 6 章独立文件
  卷02_认证与人事/ — 4 章独立文件
  ...（共 7 卷 + 附录 A~D）

源码全书/      ← 自动生成（55 个 .py.md 文件，每个模块独立）
```

### 铁律（违反则任务失败）

1. **不改 core/ 下任何文件**，包括 cosmic.py、agent.py、data.py、llm_client.py、voice.py
2. **不改已有方法的签名**（参数名、参数顺序、返回值类型），只能新增方法
3. **不改 main.py、gen_book.py、rollback_control.py**
4. **不改已有类的对外接口**，继承或组合扩展
5. 新功能通过**新增文件**或**已有文件末尾追加新方法**实现
6. 修改已有函数/方法时，保持**完全向后兼容**——原调用方不做任何改动就能正常工作
7. 所有 UI 沿用 cosmic.py 中的色彩体系和辉光效果（SPACE_VOID、ACCENT_CYAN 等），通过 import 引用，禁止在模块内重新定义同名颜色

### 风格规范
- 模块文件保持在 200-500 行
- 每个模块独立、自包含，依赖通过 import 注入
- 窗口类继承 QWidget，使用 CosmicBackground 作为底层背景
- 不使用 QTabWidget（标签页堆叠），使用轨道式独立子窗口

## 回滚管控铁律（rollback_control.py 强制约束）

**任何回滚操作必须通过 `rollback_control.py` 申请，严禁绕过。三条铁律由代码级拦截，不可绕过：**

### 铁律 1：禁止全局回滚
- 禁止 `git reset --hard`、`git checkout .`、全项目覆盖、`*`/`.`/`all` 等通配回滚
- 禁止跨模块批量回滚

### 铁律 2：逐模块申请，一次一个
- 每次只能回滚一个文件/模块，必须指定精确路径
- 调用方式：`rollback_control.rollback('modules/auth/login_window.py', '具体原因说明')`
- 原因必填，少于 5 字符会被拒绝

### 铁律 3：受保护区域禁止回滚
以下路径在任何情况下禁止回滚：
```
core/              # 核心引擎
main.py            # 应用入口
gen_book.py        # 文档生成器
rollback_control.py # 回滚管控自身
data/              # 数据库目录
项目全书/           # 项目文档
源码全书/           # 自动生成文档
```

### 回滚后必须做的事
回滚完成后，必须：
1. 在 `项目全书/版本功能地图.md` 追加回滚记录（含回滚模块、原因、记录ID）
2. 运行 `python3 gen_book.py` 重新生成源码全书

### 自检命令
```bash
python3 rollback_control.py          # 查看策略和历史
python3 rollback_control.py test     # 运行 4 项拦截测试
```

## 文档更新铁律（任务合并到代码修改中，死任务）

**任何代码修改完成后，必须一并完成以下文档更新，不得省略。**
**文档已模块化拆分——哪里改动就更新哪个分片文件，无需翻整个文档。**

### 第一步：更新项目全书（项目全书/ 目录下的分片文件）

**1.1 模块状态总表更新（项目全书/模块状态总表.md）**

涉及文件增/删/行数变化时，立即更新对应行：
- 新增文件 → 新增一行（编号如 26a/26b），标明文件路径、行数、状态、一句话功能
- 行数变化 → 更新行数列
- 文件删除 → 删除对应行

**1.2 版本功能地图追加（项目全书/版本功能地图.md）**

每次改动后在该文件顶部（最新条目位置）追加，格式：
```
### YYYY-MM-DD | 简短标题（一句话概括做了什么）

- **改了什么**：具体文件名 + 改动内容 + 改动原因（为什么改）
- **怎么改的**：技术方案简述（新增什么类/方法/表，用什么模式解决什么问题）
- **影响范围**：哪些文件关联受影响（被import的文件、联动的模块）
- **验证结果**：导入验证/功能测试是否通过
```

条目要写得让下一个接手 AI 一看就懂：
- 不要只写"修复了 bug"，要写"修复了 XX.py 中 XX 函数因空指针导致的崩溃，在调用前增加了 None 检查"
- 不要只写"优化了代码"，要写"将 XX.py 中 3 处 except: pass 改为 traceback.print_exc()"

**1.3 详细描述同步（项目全书/卷0X_xxx/ 下对应章节文件）**

如果改动涉及模块的核心逻辑变化（如新增子模块、重构架构、数据库表变更），同步更新对应章节文件。定位规则：
- `modules/auth/login_window.py` → `项目全书/卷02_认证与人事/07_蓝星登录_login_window.md`
- `modules/business/order_window.py` → `项目全书/卷03_业务运营/12_订单管理_order_window.md`
- 依此类推

### 第二步：更新源码全书（源码全书/ 目录）

**每次代码修改后运行 gen_book.py：**
```bash
python3 gen_book.py
```

gen_book.py 自动扫描项目所有 .py 文件，清空并重建 `源码全书/` 目录，为每个 .py 模块生成独立 `.py.md` 文件（含路径、行数、完整源码）。

- [ ] `项目全书/模块状态总表.md` 对应行已更新（行数/状态/备注）

### 文档编写准则

**写给下一个 AI 看，不是写给人看。** 下一个 AI 读取项目全书时需要能：
1. 一眼看清有哪些模块（模块状态总表.md）
2. 知道最近做了哪些改动（版本功能地图.md）
3. 理解改动的技术细节和原因（条目内容）
4. 找到对应的源码文件（源码全书/ 目录下按文件定位）

**禁止的写法：**
- "完善了系统"← 完善了什么？
- "修复了一些问题"← 哪些问题？怎么修的？
- "优化了性能"← 什么操作？优化了什么？

**正确的写法：**
- "修复 digital_emp_window.py 中 __init__ 硬编码 5 个预置任务导致关闭窗口后任务丢失的问题，改为 SQLite 持久化 tasks+execution_logs 两张表"
- "新增 ai_chat_styles.py（45行）抽离 INPUT_STYLE/BTN_* 等 12 个样式常量，原 ai_chat_window.py 从 656 行降至 224 行"
