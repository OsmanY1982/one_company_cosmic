### 1. OpenClaw (openclaw/openclaw) — 376K ⭐

| 来源 | Note 1/3/5/6/7/10 |
|------|-------------------|

**核心能力**：面向个人场景的跨平台 AI 助手，支持 Discord/Telegram/WhatsApp/Slack 等 25+ 消息频道，能自动操作桌面文件整理、归类等任务。

**iqra 对等能力**：
- ✅ 有完整的 Agent 执行引擎（agent_loop）
- ✅ 有文件操作工具（file_handler）
- ✅ 有桌面 GUI（PyQt5）
- ❌ 无多消息频道集成（Discord/Telegram/WhatsApp/Slack）
- ❌ 无移动端/Web 端
- ❌ 无跨平台发布能力（当前仅 macOS 桌面）

**差距分析**：OpenClaw 的核心壁垒不是"AI 能力"而是**分发渠道**——它通过 25+ 消息频道让 AI 助手无处不在。iqra 的 Agent 能力与之相当甚至更强（有完整的企业业务模块），但分发渠道仅限桌面 GUI。

**技术方案**：
1. `iqra/plugins/channel_adapters/` 新建频道适配器系统
2. 实现 3 个优先级最高的频道：`telegram_adapter.py`、`wechat_adapter.py`、`web_adapter.py`
3. 抽象 `MessageChannel` 基类（receive / send / listen / auth）
4. `core/agent_loop.py` 的消息入口从单一 GUI 扩展为多频道路由器

**优先级**：P1 — 渠道扩展是增长路径，但不紧急。先验证核心 Agent 体验。

### 2. Superpowers (obra/superpowers) — 216K ⭐

| 来源 | Note 1/3/5/6/10 |
|------|-----------------|

**核心能力**：Agent 技能框架，定义 Skill 标准格式，提供 50+ 预置技能，支持 Claude Code/Codex 等。

**iqra 对等能力**：
- ✅ 有技能系统（SKILL.md 格式，skill_loader）
- ✅ 有工具注册与按需加载
- ❌ iqra 的技能格式是私有格式，不兼容 Superpowers 标准
- ❌ 无技能市场/社区
- ❌ 技能数量远少于 50+（主要是从 Marvis 继承的技能）

**差距分析**：这是**最关键的战略差距**。Superpowers 定义的是行业标准——技能格式一旦成为事实标准，生态壁垒会极其强大。iqra 目前的 SKILL.md 是自己设计的格式，与 Superpowers 不兼容。如果 Superpowers 成为 Agent 技能的事实标准，iqra 会被生态孤立。

**技术方案**：
1. `core/skill_loader.py` 增加 Superpowers 格式兼容层：`_load_superpowers_skill(path)`
2. 建立 `iqra/skills/` 与 Superpowers 格式的双向转换器
3. P0 动作：购买/下载 Superpowers 的 50+ 技能包，运行兼容性测试
4. 评估是将自己的技能格式对齐 Superpowers，还是做双向适配

**优先级**：P0 — 这是生态层面的存亡问题。技能格式标准化不可错过。

### 3. ECC (affaan-m/ECC) — 204K ⭐

| 来源 | Note 1/3/5/6/10 |
|------|-----------------|

**核心能力**：Agent 性能优化系统。懒加载 Skills、上下文压缩记忆缓存、安全沙箱，声称节省 Token 40-60%。

**iqra 对等能力**：
- ✅ 有 context_compressor（1556行，上下文压缩）
- ✅ 有 token_saver（Token 优化）
- ✅ 有 skill_loader 按需加载
- ✅ 有安全沙箱（code_executor 隔离执行）
- ❌ 无统一的懒加载机制（Skills 全量加载而非按需）
- ❌ 无记忆缓存层（每次对话重新检索）
- ❌ 无系统性 Token 节省的量化评估

**差距分析**：iqra 有分散的性能优化组件但缺乏系统集成。ECC 的价值在于**把多个优化点整合成一套可测量的系统**——"省 Token 40-60%" 这个数字本身就是极强的营销信号。iqra 做了一样的事但没有包装成可展示的能力。

**技术方案**：
1. 新建 `core/perf_optimizer.py` 整合现有组件：context_compressor + token_saver + skill_lazy_loader + memory_cache
2. 实现 `cache/memory_cache.py` — 对话历史的 LRU 缓存层，减少重复上下文注入
3. 实现 `core/token_benchmark.py` — Token 用量对比测试工具，量化优化效果
4. 在 README 中展示 Token 节省数据（哪怕先从内部 benchmark 开始）

**优先级**：P1 — 属于"把已有能力包装好"的工作，不做也不会落后，做了能极大提升对外展示力。
